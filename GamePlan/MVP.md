# Brotato-Like Roguelite Shooter — MVP 設計計畫

> **引擎**：Unity (C#)
> **類型**：Roguelite / Arena Shooter / Auto-fire
> **參考**：Brotato — 單畫面競技場、波次制、自動射擊、Roguelite 升級

---

## 1. 功能與系統清單

| # | 功能 / 系統 | 說明 | 測試方式 |
|---|------------|------|---------|
| 1 | 角色移動 | WASD / 左搖桿控制角色八方向移動 | 手動測試：確認八方向輸入回應正確、碰撞邊界不穿牆 |
| 2 | 自動瞄準與射擊 | 自動鎖定最近敵人並開火 | 單元測試：生成假敵人，驗證鎖定切換邏輯；手動測試：觀察射擊方向 |
| 3 | 武器系統 | 至少 3 種基礎武器（手槍、散彈、SMG），可切換 | 手動測試：裝備不同武器確認射速、傷害、彈道差異 |
| 4 | 敵人生成系統 (Wave Spawner) | 按波次規則在場地邊緣生成敵人 | 自動化測試：驗證每波敵人數量遞增；手動測試：觀察生成位置與節奏 |
| 5 | 敵人 AI — 追蹤型 | 直線朝玩家移動並造成接觸傷害 | 手動測試：觀察路徑追蹤、碰撞傷害數值 |
| 6 | 生命值 / 傷害系統 | 玩家與敵人共用 HP 元件；受擊扣血、死亡銷毀 | 單元測試：扣血 → 驗證 HP 值 → 驗證死亡回呼 |
| 7 | 經驗值 & 拾取物 | 敵人死亡掉落經驗寶石，靠近自動吸附 | 手動測試：殺敵後確認寶石掉落與吸附半徑 |
| 8 | 波次間升級商店 | 波次結束後彈出 UI，隨機提供 3 個升級選項 | 手動測試：確認 UI 開啟 / 關閉、選擇後 buff 生效 |
| 9 | 屬性 (Stats) 系統 | 攻擊力、攻速、移速、HP 上限、拾取範圍等可疊加屬性 | 單元測試：疊加屬性後驗證實際數值計算正確 |
| 10 | 競技場 / 場地 | 單一封閉矩形場地 + 簡單障礙物 | 手動測試：確認邊界碰撞、障礙物擋住移動與子彈 |
| 11 | HUD（遊戲內 UI） | 顯示 HP、波次編號、剩餘敵人數、經驗條 | 手動測試：受傷 / 升級 / 換波時 UI 即時更新 |
| 12 | 遊戲流程管理 | 開始 → 波次循環 → 勝利 (存活 N 波) / 死亡 → 結算 | 手動測試：走完完整流程不卡死、結算畫面正確顯示 |
| 13 | 角色選擇（基礎） | 至少 2 個角色，各有不同初始屬性 / 武器 | 手動測試：選擇不同角色後確認初始狀態差異 |
| 14 | 物品 / 道具系統 | 被動道具提供永久屬性加成，可堆疊 | 單元測試：堆疊多個同類道具驗證數值；手動測試：商店購買後立即生效 |
| 15 | 音效與基礎 VFX | 射擊音效、受傷音效、死亡特效、拾取音效 | 手動測試：各事件觸發時有對應聲音與視覺回饋 |
| 16 | 主選單 & 暫停選單 | 標題畫面 → 開始 / 退出；遊戲中暫停 / 繼續 / 返回主選單 | 手動測試：各按鈕功能正常、暫停時遊戲時間停止 |
| 17 | 敵人種類擴充 | 遠程敵人、快速小怪、大型精英 | 手動測試：觀察不同敵人行為模式與傷害差異 |
| 18 | Boss 戰 | 每 N 波出現 Boss，擁有獨立血條與攻擊模式 | 手動測試：Boss 血條正確、攻擊模式切換正常 |
| 19 | 數據持久化 | 儲存最高波次、解鎖角色等紀錄 | 自動化測試：寫入 → 重啟 → 讀取驗證 |
| 20 | 難度曲線調整工具 | ScriptableObject 控制每波敵人種類、數量、屬性成長曲線 | 修改設定檔後 PlayMode 驗證曲線變化 |

## 2. 優先度分類

### 🔴 P1 — 可玩核心
> 沒有這些功能，遊戲完全無法執行或體驗。

| 功能 | 理由 |
|------|------|
| 角色移動 | 最基本的玩家輸入 |
| 自動瞄準與射擊 | 核心戰鬥手感 |
| 敵人生成系統 | 沒有敵人就沒有遊戲 |
| 敵人 AI — 追蹤型 | 最基礎的威脅來源 |
| 生命值 / 傷害系統 | 攻擊與受傷的基礎 |
| 競技場 / 場地 | 戰鬥發生的空間 |

### 🟠 P2 — 遊戲循環
> 讓遊戲從「技術 Demo」變成「可重複遊玩的遊戲」。

| 功能 | 理由 |
|------|------|
| 經驗值 & 拾取物 | 驅動成長迴圈 |
| 波次間升級商店 | Roguelite 核心體驗 |
| 屬性 (Stats) 系統 | 升級選項的數值基礎 |
| 武器系統 | 提供戰術選擇多樣性 |
| 遊戲流程管理 | 串接開始 → 戰鬥 → 結算 |
| HUD | 玩家需要即時資訊回饋 |

### 🟡 P3 — 內容豐富度
> 讓遊戲有足夠內容撐起多次遊玩。

| 功能 | 理由 |
|------|------|
| 物品 / 道具系統 | 增加 Build 組合深度 |
| 敵人種類擴充 | 避免戰鬥單調 |
| 難度曲線調整工具 | 設計師可快速迭代平衡 |

### 🟢 P4 — 體驗完整性
> 讓遊戲感覺「完整」而非原型。

| 功能 | 理由 |
|------|------|
| 角色選擇 | 增加重玩動機 |
| 音效與基礎 VFX | 遊戲手感大幅提升 |
| 主選單 & 暫停選單 | 基本的遊戲殼 |
| 數據持久化 | 讓玩家進度有意義 |

### ⚪ P5 — MVP 後擴充
> MVP 完成後再考慮的功能。

| 功能 | 理由 |
|------|------|
| Boss 戰 | 需要大量設計與美術資源 |
| 多人合作模式 | 網路架構複雜度高 |
| 成就系統 | 非核心玩法 |
| 更多角色 / 武器 / 道具 | 量產內容 |
| 地圖多樣化 | 不同場景與環境機制 |

## 3. 分階段功能清單

### Phase 0 — 專案基礎建設
- Unity 專案結構建立（資料夾規範、Assembly Definition）
- 輸入系統設定（New Input System）
- 基礎 ScriptableObject 架構（武器資料、敵人資料、屬性資料）
- 物件池 (Object Pool) 基礎框架

### Phase 1 — 最小可玩原型 (Playable Prototype)
- 角色移動（八方向 + 碰撞）
- 自動瞄準 → 射出子彈 → 命中扣血
- 單一追蹤型敵人
- 基礎傷害 / 死亡邏輯
- 矩形封閉場地

> **驗收標準**：玩家可在場地中移動、自動射擊、殺死敵人、被敵人殺死。

### Phase 2 — 波次循環
- Wave Spawner（波次遞增邏輯）
- 經驗寶石掉落 + 吸附
- 經驗條 → 升級觸發
- 波次間商店 UI（3 選 1 升級）
- 屬性系統（攻擊力、攻速、移速、HP 上限、拾取範圍）
- 基礎 HUD（HP 條、波次編號、經驗條）

> **驗收標準**：可連續打多波，每波間可選擇升級，屬性確實影響戰鬥。

### Phase 3 — 武器與內容
- 3 種基礎武器（手槍、散彈、SMG）
- 武器切換 / 持有機制
- 被動道具系統（至少 10 個道具）
- 2～3 種敵人類型（追蹤、遠程、快速）
- 難度成長曲線（ScriptableObject 驅動）

> **驗收標準**：不同武器手感明顯不同，道具組合影響策略，敵人種類帶來不同威脅。

### Phase 4 — 遊戲殼與手感
- 角色選擇畫面（2 個角色）
- 主選單 / 暫停選單 / 結算畫面
- 遊戲流程管理（開始 → 戰鬥 → 勝利 / 死亡 → 結算）
- 音效（射擊、受擊、死亡、拾取、UI）
- 基礎 VFX（槍口火焰、擊中閃光、死亡爆炸）
- 數據持久化（最高紀錄、角色解鎖）

> **驗收標準**：完整遊玩流程從主選單到結算無斷裂，有聲光回饋。

## 4. 開發順序（功能依賴關係）

```
Phase 0: 專案基礎建設
  │
  ▼
Phase 1: 最小可玩原型
  ├── 場地 ──────────────────────────────┐
  ├── 角色移動 ◄── 需要場地碰撞邊界      │
  ├── 生命值/傷害系統 (共用元件)          │
  ├── 自動瞄準 + 射擊 ◄── 需要移動+傷害  │
  └── 敵人 AI (追蹤型) ◄── 需要傷害系統   │
        │                                 │
        ▼                                 │
Phase 2: 波次循環                         │
  ├── Wave Spawner ◄── 需要敵人 AI + 場地 ┘
  ├── 經驗值 & 拾取物 ◄── 需要敵人死亡事件
  ├── 屬性系統 ◄── 被升級與道具依賴
  ├── 升級商店 UI ◄── 需要屬性系統 + 波次結束事件
  └── HUD ◄── 需要 HP、波次、經驗資料
        │
        ▼
Phase 3: 武器與內容
  ├── 武器系統 ◄── 需要射擊 + 屬性系統
  ├── 道具系統 ◄── 需要屬性系統 + 商店 UI
  ├── 敵人種類擴充 ◄── 需要基礎敵人 AI
  └── 難度曲線工具 ◄── 需要 Wave Spawner + 敵人資料
        │
        ▼
Phase 4: 遊戲殼與手感
  ├── 遊戲流程管理 ◄── 需要波次系統 + 勝敗條件
  ├── 角色選擇 ◄── 需要屬性系統 + 武器系統
  ├── 主選單/暫停/結算 ◄── 需要流程管理
  ├── 音效 & VFX ◄── 可隨時加入，但建議在機制穩定後
  └── 數據持久化 ◄── 需要結算數據
```

**關鍵依賴鏈**：
1. `場地 → 移動 → 射擊 → 敵人 → Wave Spawner` — 戰鬥核心鏈
2. `傷害系統 → 敵人死亡 → 經驗掉落 → 升級商店 → 屬性系統` — 成長核心鏈
3. `屬性系統 → 武器系統 / 道具系統` — 內容擴充鏈

## 5. 關鍵里程碑

| 里程碑 | 對應階段 | 可交付成果 | 驗收條件 |
|--------|---------|-----------|---------|
| **M0 — 專案啟動** | Phase 0 | 可編譯的空專案 + 輸入系統 + 基礎架構 | 專案可開啟、Input Action 綁定完成、資料夾結構建好 |
| **M1 — 第一滴血** | Phase 1 | 可移動角色、可射殺敵人 | 玩家可在場地中移動、射擊、殺死至少一隻敵人、被殺後遊戲停止 |
| **M2 — 核心循環** | Phase 2 | 完整的波次 + 升級循環 | 可連續存活 5 波以上、每波間可升級、屬性成長可感知 |
| **M3 — 內容充實** | Phase 3 | 多武器 + 多道具 + 多敵人 | 至少 3 種武器、10 個道具、3 種敵人可正常運作 |
| **M4 — MVP 完成** | Phase 4 | 完整可發布的 MVP Build | 從主選單到結算的完整流程、有音效回饋、可記錄最高分 |

## 6. 預估開發時間

> 以 **3～5 人小團隊**（程式 2、企劃 1、美術 1～2）為基準估算。

| 階段 | 預估時間 | 累計時間 | 說明 |
|------|---------|---------|------|
| Phase 0 — 專案基礎建設 | 3～5 天 | 第 1 週 | 建立專案結構、輸入系統、SO 架構 |
| Phase 1 — 最小可玩原型 | 1～2 週 | 第 2～3 週 | 核心戰鬥手感需要反覆調整 |
| Phase 2 — 波次循環 | 2～3 週 | 第 5～6 週 | UI 開發 + 數值系統較花時間 |
| Phase 3 — 武器與內容 | 2～3 週 | 第 8～9 週 | 內容製作可與程式並行 |
| Phase 4 — 遊戲殼與手感 | 2～3 週 | 第 10～12 週 | 音效/VFX 整合 + 流程串接 + QA |
| **合計** | **約 8～12 週** | | 含測試與除錯緩衝 |

**時間風險因素**：
- 戰鬥手感調整可能需要額外迭代時間
- UI/UX 設計如果沒有現成框架可能拖慢進度
- 美術資源如果採用現成素材可大幅加速

### 單波次詳細時間軸

| 時間點 | 事件 | 細節 |
|--------|------|------|
| 0s | 波次開始提示 | 畫面中央顯示「Wave N」 |
| 1s | 敵人開始生成 | 從場地四周邊緣隨機位置出現 |
| 1s～45s | 戰鬥進行中 | 敵人分批生成，不是一次全出 |
| 敵人全滅 | 波次結束 | 短暫延遲後進入商店 |
| 商店階段 | 升級選擇 + 購買道具 | 無時間限制，玩家主導節奏 |
| 玩家確認 | 進入下一波次 | 回到波次開始 |

### 難度成長機制

| 波次 | 敵人數量 | 敵人血量 | 敵人種類 | 特殊事件 |
|------|---------|---------|---------|---------|
| 1～3 | 少 (5～10) | 低 | 僅追蹤型 | 無 |
| 4～6 | 中 (10～20) | 中低 | 追蹤 + 快速 | 無 |
| 7～10 | 中 (15～25) | 中 | 追蹤 + 快速 + 遠程 | 無 |
| 11～15 | 多 (20～35) | 中高 | 全種類 + 精英怪 | 精英怪首次出現 |
| 16～20 | 大量 (30～50) | 高 | 全種類 + 精英怪 | 最終 Boss (第 20 波) |
